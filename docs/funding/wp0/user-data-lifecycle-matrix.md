# User Data Lifecycle Matrix

Owner: `DataLifecycleOwner`  
Finance co-owner: `FinanceOperationsOwner`

The plan changes current deletion semantics. Today `bridge_orders.user_id`
cascades on user deletion, while `deposit_events.user_id` becomes null and many
other user records cascade. New financial movement cannot inherit an
unconditional hard-delete path.

Retention periods below are policy classes, not invented durations: `quote TTL`,
`operational financial retention`, `security/audit retention`, and
`notification retention`. Calendar durations must come from the approved legal
and product retention policy before migration publication.

## Required migration and test gates

1. New finance tables use explicit `users.id` FKs and documented delete actions;
   unconditional cascade is rejected.
2. Merge preflight locks both users and emits the complete conflict set before
   any mutation.
3. Deletion blocks or pseudonymizes while any non-terminal operation,
   reservation, ambiguous attempt, protected market reference, or mandatory
   financial retention remains.
4. Provider references and addresses use separate keyed-HMAC lookup
   fingerprints; encryption keys/ciphertext are never lookup keys.
5. Crypto-shredding deletes the applicable encryption key/ciphertext but does
   not erase mandatory amounts, state transitions, policy revision, consent,
   transaction fingerprints, or terminal evidence.
6. Retention selector dry-run and derived/delete reports include all new
   market/event/token/notification/history/user-visible references before any
   hard-delete path is allowed.
